1. EXECUTIVE SUMMARY
================================================================================

The Project module is the core learning engine of BuildR HQ. It currently has a
rich internal flow (sprints, tasks, daily standups, quizzes, mock interviews,
leaderboards) but suffers from:

  • Too many page hops before a user can START building
  • Confusing overlap between "Browse Ideas" (seeded) and "Generate Project"
  • Redundant Fork AND Copy mechanics that serve the same purpose
  • No differentiation between platform-curated projects and user-generated ones
  • Missing AI-guided step-by-step building mode
  • Ideas page data is hardcoded in categories.ts, not in the database

This document provides a comprehensive analysis with concrete improvements.

4. SCHEMA ANALYSIS (ProjectV2 & Related Models)
================================================================================

4.1 ProjectV2 — The Core Model
─────────────────────────────────

  KEY FIELDS:
  • id, slug, title, shortDescription, description
  • technologies[], generationType, primaryLanguageOrFramework
  • difficulty (BEGINNER | INTERMEDIATE | ADVANCED)
  • visibility (PRIVATE | PUBLIC)
  • estimatedHours, includeAssessment
  • blueprintOverview, vision, targetAudience, problemSolution
  • keyOutcomes[], features (JSON), estimatedDuration
  • technicalRequirements, dataArchitecture, projectStructure, setupGuide
  • stacks (JSON), assistantEcho, assistantRaw
  • Stats: totalStarted, totalCompleted, totalSubmissions, totalViews
  • Fork tracking: isFork, forkedFromId, forkCount
  • Space: spaceId
  • University: isUniversityProject, universityId, teacherMemberId, classIds[]

  RELATIONS:
  • pages[] → ProjectV2Page
  • sprints[] → ProjectV2Sprint (which contain tasks)
  • quiz → ProjectV2Quiz
  • knowledge → ProjectV2KnowledgeBase
  • progress[] → UserProjectV2Progress
  • submissions[] → ProjectV2Submission
  • taskStatuses[] → UserTaskV2Status
  • quizAttempts[], mockSessions[], featureSuggestions[]
  • leaderboard[], resources[], errors[]
  • members[] → ProjectV2Member
  • invitations[] → ProjectV2Invitation
  • sourceIdea → ProjectIdea (linked idea)

  ⚠️ MISSING FIELDS (NEEDED):
  • isPlatformSeeded (Boolean) — Flag for platform-added projects
  • projectSource (enum: AI_GENERATED, PLATFORM_SEEDED, USER_SUBMITTED)
  • learningMode (enum: SELF_PACED, AI_GUIDED) — For step-by-step assistant

4.2 ProjectIdea — Ideas Library
────────────────────────────────

  KEY FIELDS:
  • projectTitle, projectDescription, generationType, difficulty
  • technologies[], categories[], technology
  • ideaType (PROBLEM_STATEMENT | TECHNOLOGY_SPECIFIC)
  • overview, coreRequirements[], engineeringConstraints[]
  • suggestedStacks (JSON), recruiterSignal
  • buildCount, images[], figmaLinks[], resourceLinks[]
  • stacks (JSON)
  • Blueprint linking: blueprintProjectId, hasBlueprintGenerated
  • status (PENDING | APPROVED | REJECTED)
  • isUserSubmitted, submittedById
  • upvotes, views

  ✅ GOOD: Already has blueprintProjectId for linking to pre-generated projects
  ⚠️ ISSUE: The category data in categories.ts is HARDCODED (not in DB)
  ⚠️ ISSUE: No isPlatformCurated flag on ideas

4.3 ProjectV2Sprint — Sprint Organization
──────────────────────────────────────────

  KEY FIELDS:
  • sprintNumber, name, goal, duration, orderIndex
  • createdBy, isApproved, isPersonal
  • tasks[] → ProjectV2Task

4.4 ProjectV2Task — Individual Tasks
─────────────────────────────────────

  KEY FIELDS:
  • title, description[], criteria[], hints[], badges[], tags[]
  • difficulty, orderIndex, terminalCommand
  • category, estimatedTime, checkpoints[], relatedPages[], dependencies[]
  • learningObjectives[], prerequisites[], resources (JSON)
  • testingGuidelines[], Learns (JSON), assessmentType
  • taskDetail → ProjectV2TaskDetail (sub-tasks, common errors)

5. CURRENT USER FLOW — STEP-BY-STEP WALKTHROUGH
================================================================================

FLOW A: AI-Generated Project (User Knows What to Build)
─────────────────────────────────────────────────────────
  Page 1: /projects (Landing)
    → User clicks "Generate with AI" button
    → ProjectGenerateSheet opens (multi-step wizard)
      Step 1: Choose Type (Full Stack, Frontend, etc.)
      Step 2: Title & Description
      Step 3: Difficulty Level
      Step 4: Tech Stack Selection
      Step 5: Generation (calls OpenAI Assistant)
    → On success, redirects to /projects/[slug] (detail page)

FLOW B: Browse Ideas – Technology Based
───────────────────────────────────────
  Page 1: /projects (Landing)
    → User clicks "Browse Ideas"
  Page 2: /projects/ideas (Ideas Hub)
    → User sees "Technology Based" tab (default)
    → User clicks a category card (e.g., "Web Development")
    → BOTTOM SHEET opens showing technologies
    → User clicks a technology (e.g., "React")
  Page 3: /projects/ideas/web-development/react
    → User sees list of project ideas for React
    → User clicks "Build Now" on a project idea
    → ProjectGenerateSheet opens with pre-filled title/description
    → AI generates sprints/tasks
    → Redirect to /projects/[slug]

FLOW C: Browse Ideas – Problem First
─────────────────────────────────────
  Page 2: /projects/ideas
    → User switches to "Problem First" tab
    → User sees problem statements from database
    → User clicks "Start Building" on a problem
    → ProjectGenerateSheet opens with pre-filled data
    → AI generates , redirects to /projects/[slug]

FLOW D: Project Detail & Execution
───────────────────────────────────
  Page: /projects/[slug] (Project Details)
    → Overview tab: blueprint, tech stack, key outcomes, vision, features
    → Pages tab: application pages breakdown
    → Setup Guide tab: prerequisites, env vars, installation
    → Settings tab (creator only): visibility, team, invitations
    → Creator sees "Start Building" button → starts progress tracking
    → Non-creator sees "Enroll Now" (credits) or "Fork This Project" (free)
    → After starting: milestone tracker, quick actions appear
    → "Continue Building" → redirects to /projects/[slug]/sprints

  Page: /projects/[slug]/sprints (Sprint Board)
    → Left column: task list by sprint
    → Middle/Right: task details, resources, suggestions, errors
    → Header: Daily Standup, Quiz (50% unlock), Mock Interview (75% unlock)
    → Task status: TO_DO → IN_PROGRESS → COMPLETED
    → Sprint locking: must complete previous sprint to unlock next
    → Sprint-level mock reviews

FLOW E: Fork/Enroll (Viewing Others' Projects)
──────────────────────────────────────────────
  → User finds a public project
  → Option 1: Enroll (costs 13 credits) → creates progress tracking on SAME project
  → Option 2: Fork (free) → creates a COMPLETE COPY of the project for the user
  → Both give access to sprints page

7. IDENTIFIED PROBLEMS & FRICTION POINTS
================================================================================

7.1  TOO MANY PAGE HOPS (Critical)
─────────────────────────────────────
  Current Ideas flow:
    /projects → /projects/ideas → [sheet] → /projects/ideas/[c]/[t]
    → [sheet: generate] → /projects/[slug] → Start → /sprints

  That's 4-5 page navigations + 2-3 sheet openings before actual building.

  RECOMMENDATION: Reduce to maximum 2-3 steps total.

7.2  SEEDED IDEAS OPEN GENERATE SHEET (Bad UX)
────────────────────────────────────────────────
  When a user clicks "Build Now" on a technology-based idea (hardcoded in
  categories.ts), it opens the ProjectGenerateSheet with pre-filled data.
  The user then has to go through the AI generation flow again.

  This makes no sense for platform-curated projects where we ALREADY KNOW
  what the project should look like. These should be PRE-GENERATED.

  RECOMMENDATION: Platform-curated projects should be pre-seeded with
  sprints & tasks in the database. Clicking "Start Project" should
  immediately take the user to /projects/[slug] (or directly start).

7.3  CATEGORIES DATA IS HARDCODED (Technical Debt)
──────────────────────────────────────────────────
  File: apps/main/app/(main)/projects/ideas/data/categories.ts (1015 lines)
  Contains 12 hardcoded categories with ~60+ technologies.
  This should eventually be moved to the database for admin management.

7.4  NO DISTINCTION: PLATFORM vs USER-GENERATED PROJECTS
────────────────────────────────────────────────────────
  All projects look the same. Users can't tell if a project is:
  • Curated by BuildR HQ (high quality, verified)
  • AI-generated by another user
  • Submitted as an idea by a community member

  RECOMMENDATION: Add isPlatformSeeded flag + visual badge.

7.5  FORK AND COPY — REDUNDANT (Confusing)
─────────────────────────────────────────
  Current state:
  • "Enroll" → Costs credits, tracks progress on SAME project
  • "Fork" → Free, creates a COMPLETE COPY (new project with all tasks)

  Both achieve the same goal: letting the user work on the project.
  Having both confuses users about which to choose.

7.6  1150-LINE MEGA COMPONENT
──────────────────────────────
  project-details-client.tsx is 1150 lines. While it has some extracted
  sub-components, the main file still has:
  • MilestoneTracker component
  • QuickActions component
  • All dialog/sheet state management
  • Fork, enroll, share, submit handlers
  • Team settings handlers
  • All tab rendering

7.7  IDEAS PAGE: PROJECTS NOT ACTUALLY IN DATABASE
──────────────────────────────────────────────────
  The technology-based projects on /projects/ideas are fetched from the
  ProjectIdea table, but those records often don't have pre-generated
  blueprints (hasBlueprintGenerated is false). The user has to go through
  AI generation every time.

7.8  NO AI-GUIDED BUILDING MODE
────────────────────────────────
  The platform's philosophy is "don't give solutions until asked," but
  there's no structured AI-guided mode for users who want step-by-step
  hand-holding. The AI assistant buttons exist but don't offer an
  integrated chat-based building experience.


================================================================================
8. FORK vs COPY vs ENROLL — DECISION & RECOMMENDATION
================================================================================

8.1  Analysis
─────────────

  ┌──────────────┬──────────────────────────┬────────────────────────────┐
  │   Feature    │        ENROLL            │          FORK              │
  ├──────────────┼──────────────────────────┼────────────────────────────┤
  │ Cost         │ 13 credits               │ Free                       │
  │ Data Model   │ Creates UserProgress on  │ Creates FULL COPY:         │
  │              │ SAME project             │ new ProjectV2, new tasks,  │
  │              │                          │ new pages, new sprints     │
  │ Leaderboard  │ Yes (shared)             │ No (separate project)      │
  │ Updates      │ Gets creator's updates   │ Snapshot — no sync         │
  │ Collaboration│ Can be invited to team   │ Independent                │
  │ Use Case     │ "Join this project"      │ "I want my own copy"       │
  └──────────────┴──────────────────────────┴────────────────────────────┘

8.2  Recommendation: KEEP ENROLL (rename to "Join"), REMOVE FORK
────────────────────────────────────────────────────────────────

  REASONING:
  • For PLATFORM-SEEDED projects: Enroll makes sense. The user "joins" a
    well-structured project and tracks their progress on it. They appear
    on the leaderboard. They get updates if the platform improves the project.

  • Fork creates data bloat: Every fork duplicates the entire project
    (all sprints, all tasks, all pages). With 100 seeded projects and
    10,000 users, that's potentially millions of duplicated records.

  • Fork is conceptually confusing for learners. They don't need to "own"
    the project — they need to LEARN from it.

  • The Invite feature (for teams) already covers the collaboration case.

  ALTERNATIVE: If you want a "free" option, make enrollment free for
  platform-seeded projects (since they cost nothing to generate).

  RESULT:
  • Platform-seeded projects → "Start Project" (free, creates progress)
  • User-generated public projects → "Enroll" (credits, shared leaderboard)
  • Team projects → "Invite" (existing flow)
  • REMOVE: Fork button entirely

9. SEEDED / PLATFORM PROJECTS — NEW FLOW PROPOSAL
================================================================================

9.1  Concept: Pre-Generated Project Blueprints
──────────────────────────────────────────────

  Instead of the user clicking "Build Now" → AI generates → redirect,
  platform-curated projects should ALREADY EXIST in the database as
  full ProjectV2 records with:
  • Pre-generated sprints and tasks
  • Blueprint overview, features, setup guide
  • Proper difficulty distribution (33 basic, 33 medium, 34 advanced)
  • curated by the BuildR HQ team

  The ProjectIdea already has blueprintProjectId to link to a ProjectV2.

9.2  Seeding Strategy
─────────────────────

  Target: 100 projects across categories

  ┌────────────────┬──────────┬─────────┬──────────┬──────────┐

13. AI STEP-BY-STEP ASSISTANT — DESIGN SPECIFICATION
================================================================================

13.1  Overview
──────────────

  When a user starts a project and chooses "AI-Guided" mode, they get an
  integrated AI chat panel alongside the sprint board. The AI acts as a
  mentor, guiding them task-by-task through the project.

13.2  Core Principles
───────────────────────

  1. NO CODE DUMPING: AI does not give full solutions upfront
  2. INCREMENTAL: One step at a time, wait for user confirmation
  3. SOCRATIC: Ask questions, give hints before answers
  4. CONTEXTUAL: AI knows the current task, sprint, and project context
  5. ON-DEMAND: User can ask for code, commands, or explanations when needed

13.3  Conversation Flow Design
───────────────────────────────

  STEP 1 — Project Introduction
  ─────────────────────────────
  AI: "👋 Welcome to {projectTitle}! This is a {difficulty} level
       {generationType} project. You'll be building it using
       {technologies.join(', ')}.

       Here's what you'll create:
       {blueprintOverview}

       We'll work through {N} sprints with {M} total tasks.

       Sprint 1: {sprint1.name}
       Goal: {sprint1.goal}

       Should we get started with the first task? 🚀"

  User: "Yes, let's go!"

  STEP 2 — Task Introduction (for each task)
  ──────────────────────────────────────────
  AI: "📋 Task {N}: {task.title}

       Here's what you need to do:
       {task.description.map(step => `• ${step}`).join('\n')}

       ✅ Success Criteria:
       {task.criteria.map(c => `• ${c}`).join('\n')}

       This should take about {task.estimatedTime}.

       Would you like to:
       1. Start on your own (I'll wait for questions)
       2. Get a step-by-step walkthrough
       3. See what tools/commands you'll need"

  STEP 3 — If user asks for walkthrough
  ─────────────────────────────────────
  AI breaks down each description step into sub-steps:
  - Explains the concept behind each step
  - Suggests terminal commands (without running them)
  - Shows code structure (skeleton, not implementation)
  - Waits for user to confirm they've done each sub-step

  STEP 4 — User can ask questions anytime
  ──────────────────────────────────────
  • "What does X mean?" → Concept explanation
  • "Show me the code for Y" → Code snippet (only when asked)
  • "I'm stuck on Z" → Debugging hints, then solution if needed
  • "Quiz me on this topic" → Quick quiz generation
  • "Give me the terminal command" → Command suggestion

  STEP 5 — Task Completion
  ────────────────────────
  User: "I've completed this task"
  AI: "Great! Let me verify:
       ✅ {criteria[0]} — Done?
       ✅ {criteria[1]} — Done?
       ✅ {criteria[2]} — Done?

       All good! Marking task as complete. ✨
       You're now at {progress}% completion.

       Ready for the next task? Or would you like a quick quiz
       on what you just learned?"

13.4  UI Layout for AI-Guided Mode
──────────────────────────────────

  ┌────────────────────────────────────────────────────────────────────┐
  │  Sprint Board Header: Sprint 1 — Foundation & Setup               │
  │  [Daily Standup] [Quiz 🔒 50%] [Mock Interview 🔒 75%]            │
  ├──────────┬────────────────────────┬───────────────────────────────┤
  │  Tasks   │   Task Detail          │   AI Assistant Panel          │
  │          │                        │                               │
  │ ✅ T-1   │  📋 Setup Database     │  🤖 AI Mentor                 │
  │ ✅ T-2   │  Description:          │                               │
  │ ▶️ T-3   │  • Install PostgreSQL  │  "You're working on Task 3:  │
  │ ⬜ T-4   │  • Create schema       │   Setting up the database.   │
  │ ⬜ T-5   │  • Add migrations      │                               │
  │          │                        │   Would you like me to walk  │
  │          │  Criteria:             │   you through each step?"    │
  │          │  • DB connected         │                               │
  │          │  • Tables created       │  ┌──────────────────────┐    │
  │          │  • Seeds working        │  │ Type your message... │    │
  │          │                        │  └──────────────────────┘    │
  ├──────────┴────────────────────────┴───────────────────────────────┤
  │  Progress: ████████░░░░░░░░░░░░  40% | Sprint 1/4 | Task 3/12   │
  └────────────────────────────────────────────────────────────────────┘

13.5  Implementation Approach
─────────────────────────────

  OPTION A: OpenAI Chat Completions (Recommended for V1)
  • Use GPT-4 with system prompt containing project context
  • Store conversation history in ProjectV2GuidedSession
  • System prompt includes: project data, current task, sprint info
  • Each message sent with context of task progress
  • Costs: ~$0.01-0.05 per message exchange

  OPTION B: OpenAI Assistants API (For V2)
  • Create per-project assistant with knowledge base
  • Thread-based conversation with file attachments
  • Better long-term memory and tool usage
  • Higher cost but more sophisticated

  OPTION C: Pre-Seeded Messages + AI (Hybrid)
  • For each task, pre-define the intro message and key steps
  • Use AI only for user questions and debugging
  • Cheapest option, most predictable behavior
  • Store pre-defined steps in task.guidedSteps (JSON)

  RECOMMENDATION: Start with Option C for platform-seeded projects
  (pre-seeded guided steps) and Option A for AI-generated projects.

13.6  Data Model for Guided Steps
─────────────────────────────────

  Each task can have an optional `guidedSteps` JSON field:

  {
    "introduction": "Welcome to this task! You'll be setting up...",
    "steps": [
      {
        "instruction": "First, install the required dependencies",
        "command": "npm install prisma @prisma/client",
        "explanation": "Prisma is an ORM that...",
        "verification": "Run 'npx prisma --version' to verify"
      },
      {
        "instruction": "Create the Prisma schema file",
        "codeSnippet": "// prisma/schema.prisma\ngenerator client {...}",
        "explanation": "The schema file defines your database models...",
        "verification": "Check that prisma/schema.prisma exists"
      }
    ],
    "completion": "Great! You've successfully set up the database layer.",
    "quiz": [
      {
        "question": "What is an ORM?",
        "options": ["A database", "An Object-Relational Mapper", ...],
        "correct": 1
      }
    ]
  }
