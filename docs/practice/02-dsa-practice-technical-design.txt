================================================================================
DSA PRACTICE: TECHNICAL DESIGN & IMPLEMENTATION
================================================================================

1. OVERVIEW
===========
The Data Structures & Algorithms (DSA) practice module needs to bridge the gap between mindless trial-and-error and actual conceptual mastery. 

URL: `/mock/practice/dsa/[problemId]`

2. UI & LAYOUT SPECIFICS
========================
The screen will be divided into three main columns:

A. Left Column: Problem Context
- Problem statement, constraints, example input/output.
- A toggle for "Company Tags" or "Pattern Hints" (e.g., Sliding Window, Two Pointers).

B. Center Column: Workspace
- Code editor (Monaco or similar, already used in Coderz).
- Language selector (TypeScript, Python, C++, Java).
- "Run" and "Submit" buttons. "Run" is the primary AI trigger.

C. Right Column: AI Mentor & Test Results
- Standard chat interface but enriched with Markdown.
- Voice toggle button (to enable/disable voice feedback).
- Visualizer widget (renders Mermaid.js diagrams).

3. THE "RUN" INTERACTION FLOW
=============================
When the user hits "Run", we don't just compile the code. We undergo a 3-step process:

Step 1: Test Execution (Dry Run)
Execute the user's code against basic, visible test cases using a secure execution environment or standard backend runner.

Step 2: AI Prompt generation (The "Secret Sauce")
If the code fails or is suboptimal, we package the payload for the LLM:
```json
{
  "problem": "Fibonacci Number",
  "optimalComplexity": "O(N) Time, O(1) Space",
  "userCode": "def fib(n): \n  if n<=1: return n\n  return fib(n-1) + fib(n-2)",
  "errorState": "Time Limit Exceeded for n=45",
  "instruction": "The user used brute-force recursion. DO NOT give the solution for DP or memoization. Instead: 1. Acknowledge their recursive approach. 2. Explain why it fails for large N using a markdown flowchart of the recursive tree. 3. Ask them how they might store previously calculated values."
}
```

Step 3: Response Parsing & Rendering
The AI returns a formatted markdown response containing:
- Conversational text (to be read aloud by ElevenLabs API if voice is enabled).
- A code block with the `mermaid` language tag containing the execution tree.

4. FLOWCHART & VISUALIZATION (MERMAID.JS)
=========================================
Because all AI chat content is rendered in Markdown, we can use `react-markdown` in conjunction with `rehype-mermaid` to instantly render flowcharts.

Example AI response for Fibonacci:
"You're using standard recursion. This is a great start! But look at how many times we calculate the same value...

```mermaid
graph TD
  F5[fib(5)] --> F4[fib(4)]
  F5 --> F3[fib(3)]
  F4 --> F3_2[fib(3)]
  F4 --> F2[fib(2)]
  F3 --> F2_2[fib(2)]
  F3 --> F1[fib(1)]
  style F3_2 fill:#f9f,stroke:#333,stroke-width:4px
  style F3 fill:#f9f,stroke:#333,stroke-width:4px
```

See how `fib(3)` gets evaluated multiple times? What data structure could we use to save `fib(3)` the first time we see it?"

5. PROGRESSIVE OPTIMIZATION TRACKING
====================================
The AI needs context. We will maintain a `practiceSession` state.
- Attempt 1: Brute Force -> AI explains the bottleneck.
- Attempt 2: Memoization -> AI says, "Great! This is O(N) time and O(N) space. Can we do O(1) space?"
- Attempt 3: Tabulation with 2 variables -> AI congratulates and registers the success in the database.

6. VOICE FEEDBACK INTEGRATION
=============================
When the AI generates its response, we extract the conversational text (stripping out the Markdown/Mermaid blocks) and send it to the Voice API (ElevenLabs).
The user can optionally respond via microphone: "I guess I could use an array to store the values?"
This is transcribed and appended to the chat, triggering the next AI evaluation
