================================================================================
                           1. EXECUTIVE SUMMARY
================================================================================

The Projects Module is a core feature of the Coder'z platform that enables users 
to generate, build, and track coding projects. This document analyzes the current
implementation, identifies gaps, and proposes improvements to create a unified,
cohesive experience.

KEY MODULES ANALYZED:
- Project Ideas (/projects/ideas)
- Project Generation (/projects/generate via ProjectGenerateSheet)
- Project Details (/projects/[slug])
- Submit Project Idea Sheet (user submissions)
- Project V2 (generated projects database)
- Project Idea (curated ideas database)

CURRENT DATA FLOW:
------------------

1. IDEAS FLOW:
   - User visits /projects/ideas
   - Selects a category (Web Development, AI/ML, etc.)
   - Selects a technology (React, Node.js, etc.)
   - Sees list of ProjectIdea records from database
   - Clicks "Build Now" → redirects to /projects/generate with URL params

2. GENERATION FLOW:
   - User opens ProjectGenerateSheet (from /projects or /projects/ideas)
   - Fills out form: title, description, type, stack, difficulty
   - Submits → Worker API generates project with AI
   - New ProjectV2 record created in database
   - NO LINK between ProjectIdea and ProjectV2!

3. SUBMISSION FLOW:
   - User clicks "Submit Idea" on /projects/ideas
   - Opens SubmitProjectIdeaSheet
   - Fills: title, description, type, difficulty, technology, etc.
   - Creates new ProjectIdea with status: PENDING
   - Admin approves → status: APPROVED
   - User earns 20 XP

================================================================================
                        3. IDENTIFIED ISSUES & GAPS
================================================================================

┌─────────────────────────────────────────────────────────────────────────────┐
│                           CRITICAL ISSUES                                    │
└─────────────────────────────────────────────────────────────────────────────┘

ISSUE #1: DISCONNECTED IDEAS AND PROJECTS
------------------------------------------
Problem: When user clicks "Build Now" on a ProjectIdea, a new ProjectV2 is
created, but there's NO LINK back to the original idea.

Impact:
- Cannot show "5 users built this project"
- Cannot share community solutions
- Cannot track which ideas are popular for building
- Duplicate projects created each time

Current Schema:
  ProjectIdea has: blueprintProjectId (optional, for pre-generated templates)
  ProjectV2 has: NO reference to ProjectIdea

ISSUE #2: TWO TYPES OF IDEAS NOT DISTINGUISHED
-----------------------------------------------
Problem: User wants two types of ideas:
  
  TYPE A: "Problem Statements" (Technology Agnostic)
    - Just a problem to solve
    - User chooses their own tech stack
    - Example: "Build an Event-Driven Notification Service"
    
  TYPE B: "Technology-Specific Ideas" (Current Implementation)
    - Has specific technology (React, Node.js, etc.)
    - Pre-defined stack suggestions
    - Example: "Build a React Task Manager"

Current: Only Type B is supported. Type A has no place.

ISSUE #3: NO CLONE/FORK FUNCTIONALITY
--------------------------------------
Problem: If User A generates "E-commerce Platform", and User B wants to build
the same thing, User B must:
  1. Go to ideas
  2. Find the idea
  3. Generate a new project from scratch

Better: User B should be able to "Clone" or "Fork" User A's project:
  - Start from same checkpoint
  - Get own timeline
  - Track that it's a fork

ISSUE #4: SUBMIT IDEA SHEET REQUIRES TECHNOLOGY
------------------------------------------------
Problem: Current SubmitProjectIdeaSheet requires:
  - Generation Type (required)
  - Difficulty (required)
  - Main Technology (required) ← This prevents problem statements!

Impact: Users cannot submit technology-agnostic problem statements.

ISSUE #5: NO PROJECT DISCOVERY FROM BUILDS
-------------------------------------------
Problem: If 10 users built "React Todo App", there's no way to:
  - See all 10 implementations
  - Compare approaches
  - Learn from others' solutions

The ideas page is one-directional: Idea → Generate
It should be bidirectional: Idea ↔ Projects (built from idea)

ISSUE #6: BLUEPRINT/SOURCE IDEA LINK EXISTS BUT NOT FULLY USED
----------------------------------------------------------------
Schema shows: 
  - ProjectV2.sourceIdea → ProjectIdea (via IdeaBlueprint) ✅ EXISTS!
  - ProjectIdea.blueprintProjectId → ProjectV2 ✅ EXISTS!
  - ProjectIdea.hasBlueprintGenerated ✅ EXISTS!

Current State:
  - The relation EXISTS in the schema
  - But: "Build Now" from ideas page does NOT use this link!
  - Projects are generated fresh each time
  - The sourceIdea is never set during generation

What's Missing:
  - Actually passing sourceIdeaId when generating from an idea
  - Updating buildCount on the idea when someone builds it
  - UI to show "Built from: [Idea Name]" on projects
  - "Community Builds" section on idea detail page

PROPOSAL 1: TWO-TAB IDEAS PAGE
-------------------------------

/projects/ideas
├── Tab 1: "Problem Statements" (NEW!)
│   ├── Technology-agnostic challenges
│   ├── Card shows: Title, Overview, Core Requirements, Engineering Constraints
│   ├── User clicks → Opens detail sheet with:
│   │   - Full problem description
│   │   - Suggested stacks (optional)
│   │   - "Start Building" → Opens ProjectGenerateSheet (pre-filled title/desc)
│   └── Users can submit problem statements via same sheet (just different mode)
│
└── Tab 2: "Technology Ideas" (Current)
    ├── Organized by Category → Technology
    ├── Specific tech stack ideas
    └── "Build Now" works as before

PROPOSAL 2: LINK IDEAS TO PROJECTS
-----------------------------------

Database Change:
  ProjectV2 {
    // ... existing fields
    sourceIdeaId: String?     // Link to ProjectIdea
    sourceIdea: ProjectIdea?  // Relation
    isFork: Boolean           // If forked from another ProjectV2
    forkedFromId: String?     // Original project if forked
  }

  ProjectIdea {
    // ... existing fields
    builtProjects: ProjectV2[] // All projects built from this idea
    buildCount: Int            // Denormalized count for performance
  }

PROPOSAL 3: UNIFIED SUBMIT SHEET WITH MODES
--------------------------------------------

SubmitProjectIdeaSheet:
├── Mode: "Problem Statement"
│   ├── Title (required)
│   ├── Overview (required, markdown)
│   ├── Core Requirements (checklist)
│   ├── Engineering Constraints (optional)
│   ├── Suggested Stacks (optional, multi-select)
│   └── technology = null (marks it as problem statement)
│
└── Mode: "Technology Idea" (Current)
    ├── Title (required)
    ├── Description (required)
    ├── Main Technology (required)
    ├── Additional Tech (optional)
    └── Difficulty (required)

PROPOSAL 4: PROJECT FORKING SYSTEM
-----------------------------------

When viewing a public ProjectV2:
  [Clone This Project] button
    → Creates new ProjectV2 with:
      - Same title (+ "My Version")
      - Same blueprint/tasks
      - forkedFromId = originalProject.id
      - User's own progress tracking

PROPOSAL 5: COMMUNITY BUILDS SECTION
--------------------------------------

On ProjectIdea detail page:
  "Community Builds" section:
    - Shows all ProjectV2 where sourceIdeaId = thisIdea.id
    - Cards show: User avatar, project title, completion %, stars
    - Click to view their project

MODIFIED ProjectIdea MODEL:
---------------------------

model ProjectIdea {
    id                         String   @id @default(cuid())
    
    // --- Core Fields ---
    projectTitle               String
    projectDescription         String   @db.Text
    
    // --- Problem Statement Fields (NEW) ---
    ideaType                   IdeaType @default(TECHNOLOGY_SPECIFIC)
    overview                   String?  @db.Text    // For problem statements
    coreRequirements           String[] // Checklist items
    engineeringConstraints     String[] // Technical constraints
    suggestedStacks            Json?    // { lang: [...], queue: [...], db: [...] }
    recruiterSignal            String?  @db.Text    // Why recruiters care
    
    // --- Existing Fields ---
    generationType             String
    difficulty                 String
    primaryLanguageOrFramework String?  // Optional for problem statements
    technologies               String[]
    categories                 String[]
    technology                 String?  // Optional for problem statements
    
    // ... rest of fields
    
    // --- Stats ---
    buildCount                 Int      @default(0) // How many built this
    
    // --- Relation to Built Projects ---
    builtProjects              ProjectV2[] @relation("IdeaToProjects")
}

enum IdeaType {
    PROBLEM_STATEMENT      // Technology agnostic
    TECHNOLOGY_SPECIFIC    // Current behavior
}


MODIFIED ProjectV2 MODEL:
-------------------------

model ProjectV2 {
    // ... existing fields
    
    // --- Link to Source Idea ---
    sourceIdeaId    String?
    sourceIdea      ProjectIdea? @relation("IdeaToProjects", fields: [sourceIdeaId], references: [id])
    
    // --- Fork Tracking ---
    isFork          Boolean      @default(false)
    forkedFromId    String?
    forkedFrom      ProjectV2?   @relation("ProjectForks", fields: [forkedFromId], references: [id])
    forks           ProjectV2[]  @relation("ProjectForks")
    forkCount       Int          @default(0) // Denormalized
}
